# HelmRelease chart sources

## Contents
- Pattern A — OCIRepository (preferred)
- Pattern B — HelmRepository
- Pattern C — GitRepository (rare)
- values vs valuesFrom
- Inherited defaults

The Anton repo uses three Flux source kinds for HelmRelease charts. OCIRepository is the default; the others exist for charts that have not migrated to OCI or that live in git.

## Pattern A — OCIRepository (preferred)

Use for any chart published to an OCI registry. Most modern charts (stakater, cilium, external-dns, cert-manager, …) ship this way.

```yaml
---
apiVersion: source.toolkit.fluxcd.io/v1
kind: OCIRepository
metadata:
  name: reloader
spec:
  interval: 15m
  layerSelector:
    mediaType: application/vnd.cncf.helm.chart.content.v1.tar+gzip
    operation: copy
  ref:
    tag: 2.2.3
  url: oci://ghcr.io/stakater/charts/reloader
---
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: reloader
spec:
  chartRef:
    kind: OCIRepository
    name: reloader            # MUST match OCIRepository.metadata.name
  interval: 1h
  values:
    # …chart values…
```

The `layerSelector` line is required so Flux extracts the Helm chart layer from the OCI artifact rather than treating the whole image as opaque bytes. Copy it verbatim.

## Pattern B — HelmRepository

Use for charts on a classic Helm repository without an OCI mirror. Examples in this repo: harbor, cloudnative-pg.

```yaml
---
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: harbor
spec:
  interval: 15m
  url: https://helm.goharbor.io
---
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: harbor
spec:
  interval: 1h
  chart:
    spec:
      chart: harbor
      version: 1.18.0
      sourceRef:
        kind: HelmRepository
        name: harbor
  values:
    # …chart values…
```

Note the shape difference: HelmRepository uses `spec.chart.spec.sourceRef`, NOT `spec.chartRef`.

## Pattern C — GitRepository (rare)

Use only when the chart lives in a git tree — a fork, an unreleased chart, or a chart embedded inside another repo. Almost no apps in this repo use this pattern.

```yaml
---
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-fork
spec:
  interval: 1h
  url: https://github.com/example/my-fork
  ref:
    branch: main
---
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
spec:
  interval: 1h
  chart:
    spec:
      chart: ./charts/my-app           # path inside the git repo
      sourceRef:
        kind: GitRepository
        name: my-fork
  values:
    # …chart values…
```

## values vs valuesFrom

Inline `values:` for everything that fits cleanly in YAML. Use `valuesFrom:` when a value must be sourced from a Secret or ConfigMap (typically a credential generated by an ExternalSecret).

```yaml
spec:
  values:
    auth:
      enabled: true
  valuesFrom:
    - kind: Secret
      name: my-app-credentials       # produced by an ExternalSecret in the same namespace
      valuesKey: password
      targetPath: auth.password
```

`targetPath` is dot-notation into the values tree. `valuesKey` is the field within the source Secret/ConfigMap.

## Inherited defaults

Do not set these per HelmRelease — they are injected globally from `kubernetes/flux/cluster/ks.yaml` patches:

- `install.crds: CreateReplace`
- `upgrade.cleanupOnFail: true`
- `install.remediation.retries`, `upgrade.remediation.retries`
- `upgrade.remediation.strategy: rollback`

If you find yourself adding these explicitly, check the root `ks.yaml` patches first — you are probably duplicating an inherited default.
