# Plan template

Canonical anton plan template. Mirrors `context/plans/TEMPLATE.md` — keep them in sync. The `planner` skill uses this version when authoring new plans; the in-tree copy is the human-facing reference.

Use this exactly. Don't add or reorder sections without updating both copies and `references/conventions.md`.

---

```markdown
---
status: In-progress
opened: YYYY-MM-DD
closed: null
affects: <storage|observability|databases|registries|demos|networking|security|compute|all>
intent: <concrete-need|learning|unknown>
related-adrs: []
review-by: null
---

# NNNN — Title in present tense

> One-sentence summary that makes sense in the injected active-plan index.

## Goal

One paragraph. What "done" looks like in plain language — not a task list, the outcome state.

## Acceptance criteria

Outcomes that must be true at `close done`. Not tasks — conditions. 2–5 bullets.

- [ ] Criterion 1 (e.g., "all workloads backed by Longhorn volumes")
- [ ] Criterion 2 (e.g., "runbook for single-node loss added to `docs/`")
- [ ] Criterion 3

## Tasks

Concrete next steps. Append freely as the work uncovers more. Check off as they complete.

- [ ] First next step
- [ ] Second next step
- [ ] Third next step

Use sub-headings (`### Phase 1: …`) for multi-phase plans. Keep tasks imperative ("Deploy X", not "X should be deployed") and verifiable.

## Log

Append-only timeline of decisions and surprises made *during* execution. Newest at the bottom. Format: `- YYYY-MM-DD: <one sentence>`.

- YYYY-MM-DD: Plan opened — <why now>.

## References

Pointers to ground truth — this plan does not duplicate what lives elsewhere.

- Related ADRs: 0005, 0006
- Renovate dashboard: <URL or issue number>
- Flux kustomizations: `longhorn`, `longhorn-config`
- Cluster check: `kubectl get hr -n storage`
- Memory entries this plan supersedes: `project_longhorn_rollout.md` (to be deleted once plan is in use)
```

---

## Variant — learning-intent plans

When `intent: learning`, `review-by:` is **required**, not optional. The review date is the load-bearing exit condition: when the date arrives, the plan's `close` workflow checks whether the learning goal was met and flips to `Done` or `Abandoned` accordingly. The plan's Goal section should explicitly name the exit plan ("if X is learned by <date>, close Done; else uninstall per runbook in …").

## Variant — plan driven by an ADR handoff

When a plan follows directly from an `Accepted` ADR (especially one generated by `cluster-intake-gatekeeper`), the `related-adrs:` frontmatter must cite the source ADR, and the Log's opening entry should read `- YYYY-MM-DD: Opened from ADR <NNNN>`. This lets future sessions walk back from plan → decision without searching.
