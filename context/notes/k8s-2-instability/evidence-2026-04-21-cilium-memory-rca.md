# Evidence — 2026-04-21 — cilium-agent memory RCA and intervention verification

Summary of the session-2 (2026-04-21) investigation of hypothesis #4 (cilium-cni memory pressure), the durable mechanism found, the intervention applied, and the first ~90 min of post-rollout observation.

This file is the **evidence record**. For execution state and session-by-session Log, see [`../../plans/0007-k8s-2-remote-diagnostic-rollout.md`](../../plans/0007-k8s-2-remote-diagnostic-rollout.md).

## TL;DR

- Hypothesis #4 was right about the *what* (cilium-agent memory pressure on k8s-2) and wrong about the *mechanism*. It's not "many cilium-cni helpers being OOM'd by cluster-wide pressure." It's a **self-perpetuating page-cache accumulation inside the cilium-agent cgroup that OOMKills the cilium-agent itself at the 1536 MiB limit, whose restart cycle generates more cache activity, leading to another OOMKill**.
- Evidence: the previous cilium-agent on k8s-2 has `exitCode: 137` (SIGKILL from OOM-killer) at `2026-04-20T22:45:56Z`. Pod-level WSS was 739 MiB (k8s-1 190 MiB, k8s-3 189 MiB). The 550 MiB excess was in **page cache** (166 MiB cgroup cache vs 3 MiB on peers — 55x), not RSS, not endpoints, not BPF maps.
- Intervention: `memory.limit` raised from `1536Mi` → `2560Mi` in `kubernetes/apps/kube-system/cilium/app/helmrelease.yaml` (commit `0ab84e77`). DaemonSet rolled in ~60s.
- Verification: post-rollout k8s-2 cilium-agent WSS is **122 MiB** — below peers. Cache is flat at 2 MiB across 5 ticks over 70 min. k8s-2 uptime is now **>5h** with no reboot, past the upper bound of the pre-intervention 3-4h cadence.

## Quantitative breakdown

### Pre-rollout anomaly (fresh Prometheus data, TSDB window 2026-04-19 to 2026-04-21)

**Current instantaneous readings (cilium-agent container, sampled 2026-04-21 ~02:20Z):**

| Metric | k8s-1 | k8s-2 | k8s-3 |
|---|---|---|---|
| `container_memory_rss` | 166 MiB | **148 MiB (lower!)** | 164 MiB |
| `container_memory_cache` | 3 MiB | **166 MiB (55x)** | 3 MiB |
| `container_memory_working_set_bytes` | 190 MiB | **739 MiB (3.9x)** | 189 MiB |

**Memory limit**: `cilium-agent.resources.limits.memory: 1536Mi` (same on all three nodes).

**48h trend of pod-total memory** (4h step):
- k8s-1: `552-584 MiB` — flat band
- k8s-3: `465-636 MiB` — flat band
- k8s-2: **`601-1775 MiB`** — spread of 3x, repeatedly approaches the 1536 MiB limit

### The smoking gun — OOMKilled cilium-agent on k8s-2

```
$ kubectl -n kube-system get pod cilium-7qnnq -o jsonpath='{.status.containerStatuses[?(@.name=="cilium-agent")].lastState}'
{
  "terminated": {
    "exitCode": 137,
    "reason": "Error",
    "startedAt":  "2026-04-20T17:52:35Z",
    "finishedAt": "2026-04-20T22:45:56Z",
    "message": "...incremental policy update... Processing identity update..."
  }

```

Exit code 137 = SIGKILL from the kernel OOM-killer. Pod had survived ~4h53m before the kill. `restartCount=3` for `cilium-agent` on k8s-2 at the time — consistent with a recurring OOM cycle. `node_vmstat_oom_kill{instance="192.168.1.99:9100"}` reports 7 OOM events on k8s-2 over the 48h TSDB window, zero on k8s-1 and k8s-3.

### What the memory is NOT

Things that looked plausible but were ruled out during mechanism analysis:

1. **Workload on k8s-2 — no.** `cilium endpoint list` count: k8s-1=35, **k8s-2=3**, k8s-3=36. k8s-2 has *fewer* workloads because it's cordoned. Fewer endpoints, lower BPF map entry counts, but *more* cgroup memory — that's a leak, not load.

2. **BPF map size — no.** `cilium-dbg map list` shows k8s-2's maps are the *lightest*: 7 `cilium_lxc` entries, 2 `cilium_policy_v2_*` maps, 78 `cilium_lb4_backends_v3`. Peers have 37-39 policy maps and 35+ lxc entries.

3. **Go heap — no.** `go_memstats_heap_inuse_bytes` sampled from `/metrics` on port 9962: k8s-1=127 MiB, k8s-2=129 MiB, k8s-3=107 MiB. Near identical. `go_memstats_sys_bytes` actually *lower* on k8s-2. Go runtime is not the culprit.

4. **Stdout log volume — no.** `kubectl logs --since=1h` line counts: k8s-1=12, k8s-2=14, k8s-3=40. k8s-2 is the *quietest*. The container log file is not the cache source.

5. **Kernel-charged BPF memory — unlikely.** BPF maps on newer kernels can charge to the creating process's memory cgroup, but k8s-2's maps are smaller than peers', so this can't explain more memory.

### What the memory IS

**Page cache, generated by cilium-agent's own file I/O, accumulating inside its cgroup.** Plausible sources, none individually verified but all fitting the pattern:

- BPF ELF files loaded from `/var/lib/cilium/bpf/` on agent restart — each load brings the ELF into page cache in the agent's cgroup
- State replay from `/var/run/cilium/` on restart — endpoint DB, identity cache, policy state
- Repeated `stat()` or `mmap()` activity from CNI plugin invocations for pod-lifecycle events on stuck pods (k8s-2 has 7 stuck ContainerCreating pods whose sandboxes keep generating containerd events that reach cilium-agent)

Under steady state on k8s-1 / k8s-3, page cache sits at 3 MiB — normal level for Go process file I/O. On k8s-2 it grows to 166 MiB *inside the cgroup account*, driving WSS = RSS + active-cache past the 1536 MiB limit.

### Feedback loop

```
1. initial perturbation on k8s-2 (ghost pods, stuck ContainerCreating,
   pod-lifecycle events) → cilium-agent works harder
2. cilium-agent WSS grows, approaches 1536 MiB limit
3. OOMKill (exit 137)
4. kubelet restarts cilium-agent
5. agent re-loads BPF ELFs + state files → new page cache
6. WSS climbs back toward the limit within ~4h
7. GOTO 3
```

The fact that the limit was **tight** (~2x the peer baseline at ~750 MiB) meant any perturbation could push it over. Peers on stable nodes never even approached the limit.

## Intervention

Single change: `kubernetes/apps/kube-system/cilium/app/helmrelease.yaml:146`:

```diff
-        memory: 1536Mi
+        memory: 2560Mi
```

Committed as `0ab84e77` ("fix(network): raise cilium-agent memory limit 1536Mi → 2560Mi (plan 0007)"). Pushed to `main`. Flux reconciled the HelmRelease cleanly. Cilium DaemonSet rolled all three pods in ~60s (one at a time, ~20s per pod).

### Reasoning for 2560 MiB

- Observed 48h k8s-2 pod-total peak: **1775 MiB**
- Target headroom: ≥1.5x peak → ≥2662 MiB
- Round to nearest 128-MiB boundary: **2560 MiB** (a 2048+512 cleanly-representable value)
- Peers at steady state: ~584/636 MiB → 2560 MiB is nowhere near tight for them

This is an uprated ceiling, not a "remove the limit" — we still want early-warning if cilium starts genuinely leaking on a peer, rather than the kernel OOM-killer doing silent cleanup.

## Post-rollout verification

**Post-rollout instant snapshot (fresh pods, ~6 min old at 02:45Z):**

| Metric | k8s-1 | k8s-2 | k8s-3 |
|---|---|---|---|
| `container_memory_rss` | 145 MiB | **112 MiB** | 148 MiB |
| `container_memory_cache` | 3 MiB | **2 MiB** | 2 MiB |
| `container_memory_working_set_bytes` | 167 MiB | **122 MiB (below peers!)** | 171 MiB |

The anomaly was the stateful accumulation inside the restart-looping pod, **not** inherent to k8s-2 hardware or configuration. Fresh pod = fresh cgroup = peer-baseline memory.

**Observations:**

- **Cache flat at 2 MiB across all 5 ticks.** This was the single anomalous metric pre-rollout. Now pegged at peer baseline. Strongest evidence the feedback loop is broken.
- **RSS climbed +30 MiB in the first 30 min then stabilized.** Initial agent bootstrap (policy identities, endpoint sync, service state loading). Final value 142 MiB = within peer range (k8s-1 145 MiB, k8s-3 148 MiB).
- **WSS trajectory 150→151→152→152→154** = noise. For comparison, the old pod gained ~100 MiB/hour over the same kind of window.
- **Zero cilium-agent restarts. Zero new OOM events.**
- **k8s-2 has survived past 5h uptime with no node-level reboot.** Pre-intervention reboot cadence was 3-4h; the upper bound of that cadence passed at ~01:39Z (would have fired if pattern held).

## Verdict

**Hypothesis #4 heavily supported** but not yet declared confirmed. Need more observation:

- **6h uptime** (~04:20Z, ~25 min from loop start): 1.5x old cadence upper bound → strong signal
- **8h uptime** (~06:20Z): 2x old cadence upper bound → essentially confirms the fix
- **24h uptime** (~22:20Z today): durable-fix claim
- **72h uptime** (2026-04-23 21:39Z): plan 0007 Phase 1 observation window closes — final verdict goes in the plan

If the reboot cadence holds through 24h, hypothesis #4 is confirmed, memory-limit bump is the durable fix, plan 0007 moves to Phase 5 teardown (remove Vector scaffold, keep panic sysctls). If a k8s-2 reboot fires in the next few hours, the memory bump was a correlate not the cause, and we re-open the search starting with containerd sandbox-name saturation (which still survives every reboot and keeps the 7 ContainerCreating pods stuck).
