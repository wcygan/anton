# 0007 — k8s-2 remote-only diagnostic rollout

> Three remote-only interventions to localize the k8s-2 reboot cause within the two-week no-physical-access window, plus an explicit teardown phase that removes the temporary log sink and reverts kernel cmdline while keeping the panic sysctls as permanent hygiene.

## Goal

The investigation at `context/notes/k8s-2-instability/` has narrowed the live hypotheses to three (Longhorn unmount cascade, single-unit hardware defect, kernel-config-induced silent hang) but is blocked on evidence: the MS-01 has no BMC, so the "retroactively pull SEL" path is structurally unavailable, and the current Talos kernel cmdline hides the traces we'd need to discriminate. The operator is off-site for two weeks, so the DIMM-swap test and BIOS flash — the two decisive physical experiments — are parked until 2026-05-04 at earliest.

This plan is what can be done remotely in that window. It sequences three interventions from cheapest-and-most-informative to most-infrastructure: drain Longhorn off k8s-2 to falsify or confirm hypothesis #2, arm the kernel so the next hang produces a trace, and stand up a minimum-viable in-cluster log sink to capture that trace. The plan closes when the root cause is localized and either (a) fixed with a committed change or (b) handed off to a successor plan covering the physical-access follow-up. A terminal teardown phase removes the temporary log-sink infrastructure and reverts the kernel cmdline hook; only the kernel panic sysctls are retained — those are generally useful hygiene on any Talos node.

## Acceptance criteria

- [ ] Root cause of the k8s-2 reboot pattern is localized to one of: Longhorn unmount cascade, single-unit hardware defect, or a specific kernel/subsystem fault identified from captured panic traces
- [ ] Either a durable fix is committed to `talos/`, `kubernetes/apps/storage/`, or as a new ADR, OR a successor plan is opened covering physical-access work (DIMM swap, BIOS flash) with a named activation date
- [ ] Temporary log sink infrastructure is removed from `kubernetes/apps/` and the `talos.logging.kernel` kernel cmdline arg is reverted from `talos/talconfig.yaml`
- [ ] Longhorn disk eviction on k8s-2 is cleared (or made permanent via ADR, if hypothesis #2 was confirmed and the decision is to keep k8s-2 Longhorn-drained)
- [ ] Kernel panic sysctls (`kernel.panic`, `kernel.panic_on_oops`, `kernel.hardlockup_panic`, `kernel.softlockup_panic`) remain committed in `talos/patches/global/machine-sysctls.yaml` as permanent hygiene

## Tasks

### Phase 1: Longhorn drain as falsification test — already under way organically

At plan open the baseline capture found that Longhorn has **already drained k8s-2 on its own** (node condition `Ready=False` with reason `ManagerPodDown`; 0 running replicas on k8s-2; all 8 live replicas on k8s-1 + k8s-3). The natural experiment this phase was designed to create is already happening, and reboots on k8s-2 are continuing despite zero Longhorn data to unmount. This is the cheapest possible version of the test. Phase 1 is reframed from intervention to structured observation.

Note: Longhorn namespace is `storage`, not `longhorn-system` (original authoring error; corrected 2026-04-20).

- [x] Capture baseline (2026-04-20): k8s-2 last-Ready transition `21:39:12Z` (~12th reboot); 0 running Longhorn replicas on k8s-2 (5 on k8s-1, 3 on k8s-3); 8 ghost pods in `Unknown` phase on k8s-2 (longhorn-manager, longhorn-csi-plugin, engine-image-ei at 43h; 5 seaweedfs pods at 17h)

- [ ] After **≥72h of continuous observation** (target 2026-04-23 21:39Z), record verdict: hypothesis #2 **refuted** (reboots continue at same cadence with zero k8s-2 replicas — already trending this way), **inconclusive** (cadence changed but not decisively), or (unlikely given the current signal) **confirmed** — only possible if reboots stop outright once Longhorn drain is confirmed stable

### Phase 4: Root-cause triage and decision

- [ ] Classify the captured trace: CPU hard-lockup (→ hypothesis #3, likely BIOS microcode), NVMe timeout or ECC event (→ hypothesis #2 hardware), kernel NULL-deref in a specific subsystem (→ file upstream bug + pin workaround), or something unexpected
- [ ] If trace points at a committed-change fix (sysctl, Talos version bump, workload affinity): land the fix, proceed to Phase 5
- [ ] If trace points at physical work (DIMM swap, BIOS flash, NVMe replacement): open a successor plan covering the physical-access visit, with activation date ≥ operator return. Do **not** hold this plan open waiting for physical access — close it with `Done` once the successor is opened and the escalation is recorded
- [ ] If trace is inconclusive after 2 weeks: flip this plan to `Blocked`, name the specific missing signal, and reconvene at physical-access window

### Phase 5: Teardown (mandatory — do not skip)

Removes the temporary sink and un-sticks Longhorn. The kernel panic sysctls (Phase 2 output) stay committed; everything else reverts.

- [ ] Revert `machine.install.extraKernelArgs` change in `talos/talconfig.yaml`; regenerate configs; roll per-node to drop the `talos.logging.kernel=...` arg
- [ ] Remove `talos/patches/global/machine-logging.yaml` (the `machine.logging.destinations` patch); regenerate; apply per-node non-destructively
- [ ] Delete `kubernetes/apps/observability/talos-log-sink/` directory and its kustomization entry; let Flux garbage-collect the Deployment, Service, and PVC (verify PVC deletion actually lands — may need manual finalizer clear)
- [ ] Clear Longhorn disk eviction on k8s-2 by setting `Eviction Requested: false` back on both disks in the `storage` namespace. **Skip entirely if Phase 1 tripwire never fired** — the natural drain from `ManagerPodDown` is not an explicit eviction and clears itself once Longhorn's manager pod on k8s-2 stays up. Also skip if Phase 4 verdict was "keep k8s-2 Longhorn-drained permanently" — in that case open an ADR recording the capacity/topology change
- [ ] Confirm no stray `talos-log-sink` references remain: `rg -l 'talos-log-sink|logging.destinations|talos.logging.kernel' kubernetes/ talos/`
- [ ] Confirm panic sysctls survived the teardown: `rg 'hardlockup_panic' talos/patches/global/machine-sysctls.yaml`

Key findings: **(a)** cilium-agent container RSS on k8s-2 is `1473 MiB` vs `382 MiB` on k8s-1 and `384 MiB` on k8s-3 — **~3.85x peer baseline.** **(b)** Including BPF kernel memory (pod-level), k8s-2 cilium is `1732 MiB` vs `584 MiB` / `636 MiB` — **~2.9x peer baseline.** **(c)** 48h trend (4h step): k8s-1 tight range `552-584 MiB`, k8s-3 tight range `465-636 MiB`, **k8s-2 spread `601-1775 MiB`** — i.e. the anomaly is a cilium-on-k8s-2-specific growth, not a cluster-wide pattern. **(d)** `node_vmstat_oom_kill{instance="192.168.1.99:9100"}` over 48h shows `7` OOM events on k8s-2 (zero on k8s-1 and k8s-3). **(e)** The 21:29:02Z cilium-cni 6-process OOM event from session 1's dmesg is NOT visible in TSDB because Prometheus was zombie during that window; the 7 OOM events captured here are post-recovery and/or earlier-in-window. **(f)** memory-available on k8s-2 is currently 89.93 GiB (plenty) so the node isn't globally starved — the OOM-killer pressure is cgroup-scoped, consistent with the cilium pod hitting its memory limit rather than node-level OOM. **Hypothesis #4 is now the strongest-supported candidate** by quantitative evidence; hypothesis #2 (Longhorn cascade) is trending refuted (no re-scheduling, reboots continued through drain); hypothesis #3 (silent kernel hang) is partially armed but undetectable without CPU-lockup detectors; hypothesis "single-unit hardware defect" remains alive but parked until operator returns for the DIMM swap.
- 2026-04-21: **Session 2 — hypothesis #4 mechanism dug out.** Follow-on read-only inspection via `cilium-dbg` and Prometheus memory-type breakdown revealed the shape of the anomaly more precisely. **(a) RSS is NOT the story.** `container_memory_rss{container="cilium-agent"}` reads k8s-1=166 MiB, k8s-2=**148 MiB**, k8s-3=164 MiB — k8s-2 is actually *lower*. The earlier "3.85x" signal using `sum by (container)` was aggregating two timeseries per container (likely current + still-retained historical pod). **(b) The anomaly is in page cache.** `container_memory_cache{container="cilium-agent"}` reads k8s-1=3 MiB, k8s-2=**166 MiB**, k8s-3=3 MiB — **55x peer baseline.** Working set (RSS + active cache) = k8s-1 190 MiB / k8s-2 **739 MiB** / k8s-3 189 MiB. **(c) Endpoints are FEWER on k8s-2, not more.** `cilium endpoint list` count: k8s-1=35, **k8s-2=3**, k8s-3=36 — consistent with the k8s-2 cordon preventing scheduling. BPF map entry counts are all lighter on k8s-2 than peers. So the page-cache anomaly is NOT driven by workload state; it's driven by something in the cilium-agent's own lifecycle. **(d) The smoking gun — previous cilium-agent on k8s-2 was OOMKilled.** `kubectl get pod cilium-7qnnq -o json` shows `.status.containerStatuses[cilium-agent].lastState.terminated = {exitCode: 137, reason: Error, finishedAt: 2026-04-20T22:45:56Z, startedAt: 2026-04-20T17:52:35Z}`. Exit code 137 = SIGKILL from the kernel OOM-killer. Pod memory limit is **1536 MiB** (`cilium-agent.resources.limits.memory: 1536Mi`); the 48h trend showed k8s-2 spent extended time above 1000 MiB. **(e) Log volume is NOT the cache driver.** Last 1h: k8s-1=12 lines, k8s-2=14 lines, k8s-3=40 lines — k8s-2 is quietest. So the cache isn't from stdout/stderr capture. The remaining plausible generators are BPF ELF page-cache churn from repeated agent restarts, and state-file re-reads on each restart. **(f) Feedback loop hypothesis:** once k8s-2's cilium-agent OOMs once, it restarts, re-loads its BPF programs / state files / endpoint DB → each reload generates page-cache activity → WSS climbs back toward the 1536 MiB limit within ~4h → OOM again. The initial trigger of the first OOM may be unrecoverable (pre-Prometheus-recovery window), but the cycle perpetuates because the memory limit is tight and k8s-2 has additional perturbation sources (ghost pods, stuck `ContainerCreating` pods) that keep cilium-agent busy with pod-lifecycle events. **(g) What this means for the node reboots:** a cilium-agent OOMKill doesn't directly reboot a node, but it disrupts every pod-creation and endpoint-management path on the node, and the preceding memory-pressure window is where the 21:29:02Z "six cilium-cni children OOM'd in 1ms" pattern lives. This lines up with the 3-4h reboot cadence the instability notes documented. **Next-session decisions now data-informed:** (1) **bump cilium-agent memory limit** to 2560 MiB (cilium chart's `resources.limits.memory` override) as the lowest-risk intervention — breaks the OOM loop immediately, doesn't require node reboot, reversible; (2) **`talosctl reset` on k8s-2** to clear the ghost-pod / containerd-sandbox-saturation state that's keeping cilium-agent busy — decisive but destructive, requires etcd quorum confirmation; (3) **activate Phase 3 log sink** to capture any future cilium-agent OOM fingerprint with full context.
