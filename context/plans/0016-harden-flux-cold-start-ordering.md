# 0016 — Harden Flux dependsOn graph for cluster-wide cold-start

> Cope-side complement to plans 0013 / 0014 / 0015: tighten Flux ordering so a simultaneous 3-node reboot reconciles deterministically without operator intervention.

## Goal

Anton's MS-01 nodes have demonstrated a recurring failure mode of simultaneous (or sequential within ~30 min) silent reboots; the cause-side investigation lives in plans 0013/0014/0015. Even when the cause is fixed, the cluster will continue to cold-start at random intervals (planned reboots, power events, kernel upgrades). Today only 7 of 37 Flux Kustomizations declare `dependsOn` (~19% coverage), and 3 of those edges are weak (`wait: false` on the upstream means the dep waits for `ks-applied` rather than `Ready`). On a cold start, reconcile order relies on Flux's 1h retry interval to eventually converge — every retry is a CRD race, an ExternalSecret-not-resolved Secret, an HTTPRoute-without-Gateway. Done state for this plan: every consumer that needs a CRD, secret store, or platform Gateway declares an explicit `dependsOn` to the upstream that owns it; every platform-layer Kustomization that downstreams depend on uses `wait: true` (or `healthChecks`); a deliberate 3-node cold-start drill recovers without operator intervention and the recovery time is captured by betty's off-cluster TSDB (plan 0014).

- 2026-05-07 (rejection of Phase 3 #6 — Cilium/CoreDNS Helmfile migration): User reframe — "would this help or hurt cluster setup-from-scratch?" — flipped the answer. The Helmfile bootstrap is anton's well-trodden cluster-template path; it worked during the 2026-04-10 cluster reset (ADR 0001), it has dense community precedent, and each step is `kubectl`-debuggable + re-runnable from the failed step. Migration to Flux-managed Cilium + CoreDNS would force a two-phase Helm-adoption dance during fresh-cluster bootstrap: Helmfile installs default-config Cilium so Flux can come up; Flux's HelmRelease then has to adopt the existing release via `meta.helm.sh/release-name` matching and take over values. We just used this exact mechanism for PR #283's dragonfly migration and remember the care needed with `helm.sh/resource-policy: keep` to avoid cascade-deletes — doing it for CNI during cluster bootstrap, when there's no DNS / no metrics / no log aggregation, is the failure mode to avoid. Risk asymmetry: Helmfile-bootstrap failures are diagnosable via plain `kubectl` against an otherwise healthy API server; Flux-adoption-during-bootstrap failures hit when Flux itself isn't healthy and `flux get` / `flux logs` may be unavailable. Cope-side payoff is small: Cilium is a DaemonSet that restarts in ~30s alongside kubelet on a node reboot; Helmfile timing only affects fresh-cluster install (a once-per-rebuild event, not recurring cope). Only steady-state benefit (single config source, Renovate via Flux, flux-local CI coverage) matters at chart-bump frequency (~4-6×/yr) — too small to justify making bootstrap-time fragility worse. Decision: reject; don't revisit unless a real cope incident pinpoints Cilium or CoreDNS bootstrap timing as the recovery floor, OR Cilium config tweaks become frequent enough that dual-source drift bites.
