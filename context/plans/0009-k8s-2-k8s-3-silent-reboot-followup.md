# 0009 — localize the k8s-2 silent-reboot root cause

> Successor to plan 0007 for root-cause capture. Rejoin execution moved to plan 0010 after the 2026-04-24 correction: k8s-3's 04-23 reboot was operator-initiated, and the remaining unexplained silent reboots are both k8s-2.

## Goal

Localize the k8s-2-only silent reboot mechanism: two unexplained events remain, `2026-04-21T20:34Z` and `2026-04-23T20:47Z`. The former k8s-3 cascade framing is retracted; plan 0006 records the k8s-3 `2026-04-23T20:45Z` reboot as an operator-initiated Harbor acceptance test. Plan 0007 narrowed the suspect list but could not localize root cause because the failure signature — silent cold reboot, no kernel panic, no MCE, no oops — is invisible to every in-band channel except a kernel-side trace. Close this plan when **either** (a) root cause is localized to a specific layer (kernel silent hang, kube-apiserver/watch pressure, CNI admission fallout, NIC/firmware, or k8s-2 chassis variance) with a committed fix, **or** (b) the layered defenses are in place AND 14 consecutive days of stable uptime are observed. Plan 0010 owns staged k8s-2 rejoin execution.

### Phase 4 — Root-cause triage (re-scoped 2026-04-24)

**Re-scoped after the 2026-04-24 measurement-source correction.** The original Phase 4 was a controlled re-admission experiment framed as "confirm-or-falsify cilium cgroup step-increase." That framing is now dead: actual cilium-agent memory is 5-6% of the 3584 Mi limit across all 3 nodes, so there is no cilium-cgroup hypothesis left to test. The re-admission experiment would still exercise the pod-churn trigger, but we'd be spending a deliberate reboot risk to test a hypothesis that's already been reduced.

**New Phase 4 posture:**

1. **Don't provoke a reboot yet.** Netconsole is live (Phase 1). Wait for the next natural reboot to produce a kernel-side trace — that's the asymmetric-bet data we've been unable to capture for 3 events in a row. A controlled re-admission can still make sense later, but only after either (a) a natural event produces a trace, or (b) post-Phase-5 BIOS flash data discriminates microcode-vs-software.
2. **Do the cheap read-only investigations now** (Phases 4a, 4b below) that don't require a reboot.
3. **When Phase 5 completes and we have post-flash data** OR when a natural event fires: revisit whether a controlled re-admission experiment still has information value, using post-correction hypotheses (distributed pod-churn trigger, NIC hang, kernel silent hang).
