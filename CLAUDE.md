# CLAUDE.md

Guidance for Claude Code when working in the **anton** repo — a Talos Kubernetes homelab cluster.

## What this is

Talos Linux cluster managed by GitOps. Makejinja renders Flux, Talos, and bootstrap manifests from YAML in `templates/` into `kubernetes/`, `talos/`, and `bootstrap/`. Cluster runs Cilium CNI, Flux CD, cert-manager, external-dns, External Secrets Operator, and a Cloudflare tunnel. `cluster.yaml`, `nodes.yaml`, and `talenv.yaml` are the source of truth.

## Directory map

- `kubernetes/apps/` — Flux-managed HelmReleases and Kustomizations (3-file pattern per app)
- `kubernetes/components/` — Core cluster components (Cilium, cert-manager)
- `kubernetes/flux/` — Flux system + GitRepository sources
- `talos/` — Talos machine configs generated by talhelper from `talconfig.yaml`
- `bootstrap/` — Helmfile manifests applied before Flux takes over
- `templates/` — Makejinja source templates; custom plugin at `templates/scripts/plugin.py`
- `scripts/` — Shared shell utilities called by Task targets (extend `scripts/lib`, don't duplicate)
- `.taskfiles/` — Task targets grouped by subsystem
- `docs/` — Long-form notes (second-domain setup, etc.)
- `context/adrs/` — immutable architectural decision records (MADR + anton extensions); index injected at session start
- `context/hardware.md` — node hardware inventory (MS-01, RAM, part numbers)

## Specialized work: prefer skills and agents first

Before writing code from scratch for a common task, check whether a skill or subagent already owns it. These carry focused context and accrue memory across sessions.

**Skills** (`.claude/skills/`, invoked via the `Skill` tool):

- `cluster-intake` — intake gate before adding a new infra/system component; hard vetoes + weighted score + removal-graveyard check; returns add/defer/reject and hands off to `adr`
- `adr` — author/list/supersede architectural decision records in `context/adrs/`; the durable home for `cluster-intake-gatekeeper` verdicts and any other forward-looking decision
- `add-flux-app` — scaffold a new Flux app (3-file pattern + ExternalSecret + HTTPRoute)
- `expose-service` — HTTPRoute on `envoy-internal`/`envoy-external` + `DNSEndpoint` for secondary domains
- `debug-flux-reconciliation` — stuck Kustomization / HelmRelease / SOPS / postBuild triage
- `anton-cluster-health` — "is the cluster ok?" walk above the Talos layer
- `talos-inspect` — Talos-layer read-only triage (disks, network, health)
- `add-or-replace-node` — node add / replace / recovery
- `upgrade-talos-or-k8s` — rolling Talos or Kubernetes upgrade
- `rotate-credential` — age key + SOPS re-key, Flux deploy key, 1Password token, Cloudflare token
- `anton-repo-conventions` — Flux / postBuild / SOPS vs ESO reference
- `anton-remote-access` — talosctl/kubectl off-LAN via Tailscale MagicDNS
- `anton-upgrade-audit` — Renovate PR triage and tiered merge order

**Subagents** (`.claude/agents/`, invoked via the `Agent` tool):

- `cluster-triage` — read-only OS → CNI/DNS → Flux → platform → apps ranked punch list
- `flux-debugger` — ordered diagnostic path for stuck Flux syncs
- `flux-app-author` — writes the 3-file Flux pattern; does not apply to the cluster
- `talos-operator` — high-stakes node work; one node at a time, etcd-quorum gated
- `credential-rotator` — ordered rotation with preconditions, verification, rollback
- `conventions-linter` — fast pre-commit convention check (haiku)
- `upgrade-auditor` — Renovate PR triage, tiered merge plan, deprecation audit; accrues chart-specific memory
- `cluster-intake-gatekeeper` — read-only intake gate for new components; runs `cluster-intake` end-to-end, hands every verdict (add/defer/reject) to the `adr` skill, then to `flux-app-author` on accept

Nested CLAUDE.md files for deeper per-subsystem guidance: `kubernetes/apps/CLAUDE.md`, `kubernetes/apps/network/CLAUDE.md`, `bootstrap/CLAUDE.md`, `scripts/CLAUDE.md`, `.taskfiles/CLAUDE.md`.

## Essential commands

```sh
task configure                           # Render + validate + encrypt SOPS files
task reconcile                           # Force Flux to pull latest Git state
task talos:generate-config               # Regenerate Talos configs from talconfig.yaml
task talos:apply-node IP=<ip> MODE=auto  # Apply machine config to one node
task talos:upgrade-node IP=<ip>          # Upgrade Talos on one node
task talos:upgrade-k8s                   # Upgrade Kubernetes version (after editing talenv.yaml)
flux get ks -A                           # Kustomization status
flux get hr -A                           # HelmRelease status
```

Always run `task configure` before committing — it encrypts all `*.sops.*` files and validates manifests. Full task reference: `.taskfiles/CLAUDE.md`.

## Secrets: SOPS + External Secrets Operator

Two systems coexist, chosen by audience:

- **SOPS** for bootstrap / infrastructure secrets (age key at `age.key`, rules in `.sops.yaml`, files match `*.sops.*`). Used for anything that must exist before ESO is up.
- **External Secrets Operator** for application secrets. Pulls from the 1Password vault `anton` via the `onepassword-connect` `ClusterSecretStore`. Prefer ESO for new apps.

See `add-flux-app` skill for ExternalSecret templates and `anton-repo-conventions` skill for the SOPS-vs-ESO decision.

## Kubectl context

The cluster is accessed remotely via the Tailscale operator proxy. The active kubectl context is `tailscale-operator.<tailnet-name>.ts.net`, **not** `admin@anton`. This is the expected context for all kubectl, flux, and helm commands.

The `guard_k8s_context.py` hook resolves the expected context dynamically via `tailscale status --json` (reading `MagicDNSSuffix`), so it matches `tailscale-operator.<suffix>` without committing the literal tailnet name. If Tailscale is unavailable it falls back to `admin@anton`. Read-only subcommands (`get`, `describe`, `logs`, etc.) pass regardless of context. Override via `ANTON_KUBE_CONTEXT` if needed.

## Hard rules

- **Never** commit the literal tailnet name. Use the placeholder `<tailnet-name>.ts.net` in every committed file.
- **Never** commit unencrypted `*.sops.*` files. Verify with `find . -name '*.sops.*' -exec sops filestatus {} \;` before commits.
- **Never** log secret values in scripts — log file paths only.
- **Never** run destructive operations without explicit user confirmation: `task talos:reset`, `task template:reset`, `talosctl reset`, `kubectl delete namespace`.
- Verify kube/talos context before any mutating command. Prefer Tailscale MagicDNS hostnames (`k8s-1`, `k8s-2`, `k8s-3`) and the repo-generated `./talos/clusterconfig/talosconfig` so commands work off-LAN.
- Prefer read-only verbs when investigating: `kubectl get`/`describe`, `talosctl get`, `--dry-run=client`.
- Documentation (`.md`) must be valid UTF-8; never write control characters into docs.

## Gateway and DNS (one-liner)

`envoy-internal` serves cluster-internal HTTPRoutes via `cluster_gateway_addr`; `envoy-external` serves public HTTPRoutes via `cloudflare_gateway_addr`; `k8s_gateway` provides split-horizon DNS at `cluster_dns_gateway_addr`. **Apps on secondary domains require an explicit `DNSEndpoint` resource** — HTTPRoute annotations alone will not work. Details: `docs/docs/notes/adding-a-2nd-domain.md` and `kubernetes/apps/network/CLAUDE.md`.

## Development loop

1. Edit `cluster.yaml` / `nodes.yaml` / `talenv.yaml` or a manifest under `kubernetes/`.
2. Run `task configure` to render, validate, and encrypt.
3. Commit and push. Flux auto-syncs, or run `task reconcile` to force.
4. For Talos or node-level changes, hand off to the `talos-operator` subagent.
